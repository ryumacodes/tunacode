## Flow Maps

### 1. Model/setup flow
1. UI loads provider/model metadata through `core/ui_api/configuration.py`.
2. `configuration.py` delegates to `tunacode.configuration.models` and `tunacode.configuration.pricing`.
3. UI persists selections through `core/ui_api/user_configuration.py`.
4. `user_configuration.py` delegates to `tunacode.configuration.user_config`.

Observed call chains:
- `src/tunacode/ui/screens/setup.py:38` → `src/tunacode/core/ui_api/configuration.py:53-55` → `src/tunacode/configuration/models.py:132-139`
- `src/tunacode/ui/screens/model_picker.py:143` → `src/tunacode/core/ui_api/configuration.py:48-50` → `src/tunacode/configuration/models.py:158-186`
- `src/tunacode/ui/commands/model.py:116` → `src/tunacode/core/ui_api/configuration.py:73-75` → `src/tunacode/configuration/models.py:312-335`
- `src/tunacode/ui/commands/model.py:123` → `src/tunacode/core/ui_api/user_configuration.py:35-37` → `src/tunacode/configuration/user_config.py:271-284`

### 2. Request/tool-result flow
1. `TextualReplApp._process_request()` packages the selected model via `ModelName` and passes callbacks into `process_request()`.
2. `build_tool_result_callback()` creates a UI callback typed via `ToolResultCallback`.
3. Tool results are posted as `ToolResultDisplay` messages.
4. `TextualReplApp.on_tool_result_display()` renders a panel and, for file-edit tools, triggers LSP refresh/context updates.

Observed call chain:
- `src/tunacode/ui/app.py:339,342-351` → `src/tunacode/core/agents/main.py:744-764`
- `src/tunacode/ui/repl_support.py:201-223` → `src/tunacode/ui/widgets/messages.py:31-45` → `src/tunacode/ui/app.py:470-489`
- `src/tunacode/ui/app.py:482-489` updates edited-file state and calls `update_lsp_for_file()`

### 3. Message content/token flow
1. UI asks `core/ui_api/messaging.py` for content extraction or token estimates.
2. `messaging.py` delegates to `tunacode.utils.messaging`.
3. Utilities canonicalize message inputs when needed.

Observed call chain:
- `src/tunacode/ui/app.py:522,540` → `src/tunacode/core/ui_api/messaging.py:17-26` → `src/tunacode/utils/messaging/adapter.py:471-477`
- `src/tunacode/ui/app.py:656,658` → `src/tunacode/core/ui_api/messaging.py:29-36` → `src/tunacode/utils/messaging/token_counter.py:60-66`
- `src/tunacode/ui/screens/session_picker.py:164,166` uses the same `get_content()` path for preview strings

### 4. File autocomplete flow
1. `FileAutoComplete` lazily constructs `core.ui_api.file_filter.FileFilter`.
2. `ui_api.FileFilter` injects default ignore patterns and limits.
3. Inherited infrastructure `complete()` walks the filesystem and returns matching paths.

Observed call chain:
- `src/tunacode/ui/widgets/file_autocomplete.py:26,48` → `src/tunacode/core/ui_api/file_filter.py:13-22` → `src/tunacode/infrastructure/file_filter.py:137-165`

### 5. LSP status flow
1. UI calls `get_lsp_server_info(filepath)` after a file-edit tool completes.
2. `lsp_status.py` derives extension, looks up `SERVER_CONFIG`, and probes for an installed server command.
3. UI updates the resource bar with `server_name` and `available`.

Observed call chain:
- `src/tunacode/ui/app.py:746-750` → `src/tunacode/core/ui_api/lsp_status.py:27-58` → `src/tunacode/tools/lsp/servers.py:11,63,77`

### 6. Session/update/path flow
1. UI lifecycle and commands call `core/ui_api/system_paths.py`.
2. `system_paths.py` delegates to `tunacode.configuration.paths`.

Observed call chains:
